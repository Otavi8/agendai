Leitura recomendada

Use esta apostila como mapa, não como decoração.

1. Leia o conceito

Navegue com as setas do teclado, espaço, PageUp/PageDown ou pelos botões no canto inferior.

Visão geral

Este repositório é uma fábrica de agents.

Você não começa do zero. O projeto já entrega login, sessões, API, memória, estado, observabilidade, guardrails, frontend e exemplos de agentes.

O que você troca

  • O domínio do agente.
  • O prompt em `system.md`.
  • As tools que ele pode usar.
  • A rota e os DTOs específicos.

O que o harness mantém

  • Autenticação JWT e sessões.
  • Rate limit por endpoint.
  • Memória longa com mem0 + pgvector.
  • Checkpoints LangGraph em Postgres.
  • Métricas, logs e tracing.
Mapa mental

O modelo é só uma peça. O agent é o sistema ao redor.

FrontendLogin, chat, painel de fontes e streaming.
FastAPIRotas, DTOs, autenticação, rate limit e SSE.
AgentLangGraph ou Deep Agents com prompt, tools e estado.
CoreMemória, checkpoint, guardrails, middleware, métricas.
DadosPostgres, pgvector, banco externo, pasta sandbox, MCP.
Usuário -> frontend -> /api/v1/... -> agent.agent_invoke()
       -> LLM + tools + memória + checkpoint
       -> resposta normal ou stream -> frontend
Vocabulário essencial

Termos que aparecem o tempo todo

Parâmetros e configuração

Parâmetros são decisões que vêm do ambiente, não do código espalhado.

A fonte central é src/app/core/common/config.py. Ele lê `.env`, define defaults e expõe `settings` para o resto do app.

Arquivos de ambiente

  • .env.example: modelo seguro para copiar.
  • .env.development: desenvolvimento local.
  • .env.staging: homologação.
  • .env.production: produção.

Parâmetros importantes

  • OPENAI_API_KEY, DEFAULT_LLM_MODEL, MAX_TOKENS.
  • JWT_SECRET_KEY e expiração do token.
  • POSTGRES_* para banco e pgvector.
  • RATE_LIMIT_ENDPOINTS para limitar rotas.
  • MCP_ENABLED e MCP_HOSTNAMES_CSV.

Nunca coloque segredo direto no código. A apostila chama de parâmetro tudo que deve mudar por ambiente sem editar a lógica do agent.

Documentos de orientação

Três arquivos dizem como pensar antes de alterar código.

AGENTS.md

Regras críticas para qualquer agente que mexa no projeto: imports no topo, structlog, rate limit, plano antes de código e checkboxes.

CODEX.md

Manual prático do repo para o Codex: mapa, comandos, criação de agentes, padrões e armadilhas.

CONTEXT.md

Espaço para transformar uma ideia vaga em produto: usuário, problema, dados, guardrails, regras e decisões.

Ao criar um agent novo, não comece pelo LLM. Comece preenchendo contexto, objetivo, fontes de dados e limites.

ADR

ADR é memória arquitetural.

ADR significa Architecture Decision Record. É um registro curto para decisões difíceis de reverter, como banco, provedor de LLM, estratégia de memória, isolamento de tenants ou abordagem de sandbox.

Quando criar

  • Quando a decisão muda arquitetura.
  • Quando existem alternativas reais.
  • Quando alguém no futuro perguntaria "por que fizemos assim?".
  • Quando o custo de voltar atrás é alto.

Modelo simples

# ADR-001: Usar pgvector para memória

## Contexto
Precisamos de busca semântica por usuário.

## Decisão
Usar PostgreSQL + pgvector via mem0.

## Consequências
Menos serviços externos, mas Postgres vira peça crítica.

## Alternativas
Chroma, Pinecone, Supabase Vector.

O projeto cita `docs/adr/` como destino para ADRs, mas a pasta ainda pode ser criada quando a primeira decisão importante aparecer.

Skills

Skill é conhecimento especializado carregado na hora certa.

No `.codex/skills`

São skills para o Codex trabalhar melhor: LangGraph, Deep Agents, RAG, middleware, persistência, dependências e execução paralela.

No `text_to_sql/skills`

São skills do próprio agent de SQL: uma para explorar schema e outra para escrever queries.

skills/
  query-writing/
    SKILL.md
  schema-exploration/
    SKILL.md

A diferença central: `AGENTS.md` é instrução geral sempre presente; `SKILL.md` é instrução específica que só deve entrar quando a tarefa pede.

src/app/agents

O projeto vem com quatro referências de agent.

chatbot

Referência LangGraph explícita: nós, tools, guardrails, memória, streaming e checkpoint.

data_agent

Deep Agent por sessão para consultar banco read-only, pasta em sandbox e memória do usuário.

Deep Agent de demonstração com SQLite Chinook, tools SQL e skills de schema/query.

open_deep_research

Fluxo de pesquisa com supervisor, researcher subgraph, clarificação e relatório final.

Para estudar primeiro: leia `chatbot` se quiser entender LangGraph; leia `data_agent` se quiser entender um caso mais parecido com produto.

LangGraph

LangGraph transforma o agent em um grafo de execução.

No `chatbot`

  • StateGraph(GraphState) cria o grafo.
  • input_guardrail valida a entrada.
  • chat chama o modelo.
  • tool_call executa ferramentas.
  • output_guardrail valida a saída.

Termos-chave

  • Node: função que faz trabalho.
  • Edge: caminho entre nós.
  • Command: atualiza estado e decide próximo nó.
  • END: finaliza execução.
  • compile(): torna o grafo executável.
input_guardrail -> chat -> tool_call -> chat -> output_guardrail -> END
Deep Agents

Deep Agents é uma camada mais pronta para tarefas longas.

Enquanto LangGraph te dá controle fino do grafo, Deep Agents entrega planejamento, filesystem, skills, subagents e memória com menos código de orquestração.

Quando usar

  • Tarefas multi-etapas.
  • Consulta a arquivos e ferramentas.
  • Especialização por skills.
  • Possível delegação a subagents.

No projeto

  • text_to_sql usa create_deep_agent().
  • data_agent monta tools conforme fontes conectadas.
  • Skills aparecem em text_to_sql/skills.
  • Backend de filesystem pode ser sandboxado.
FastAPI e rotas

O agent vira produto quando ganha endpoint.

Rota

Arquivos em src/app/api/v1/ recebem requests, validam sessão, aplicam rate limit e chamam o agent.

DTO

Arquivos em src/app/api/v1/dtos/ definem modelos Pydantic para entrada e saída.

@router.post("/query", response_model=DataQueryResponse)
@limiter.limit(settings.RATE_LIMIT_ENDPOINTS["data_agent"][0])
async def query_sources(
    request: Request,
    body: DataQueryRequest,
    session: Session = Depends(get_current_session),
) -> DataQueryResponse:
    ...

Regra do projeto: toda rota precisa de rate limit, type hints e modelos Pydantic adequados.

Memória e checkpoints

O agent precisa lembrar em dois níveis.

Checkpoint

Estado da conversa por sessão/thread, via AsyncPostgresSaver. Serve para continuar o grafo e recuperar histórico.

Memória longa

Fatos úteis sobre o usuário, via mem0 + pgvector. É buscada por similaridade e atualizada em background.

checkpoint: "o que aconteceu nesta thread?"
memória longa: "o que vale lembrar deste usuário?"
Guardrails e middleware

Nem tudo deve chegar ao modelo. Nem tudo que sai deve ir ao usuário.

CamadaPapel
guardrails/Filtro de conteúdo, PII, safety check e nós de entrada/saída para grafos.
middleware/Pipeline em volta do agent: logging, erros, memória, sumarização, trimming e guardrails.
context/Ajuda a reduzir contexto, truncar chamadas de tools e resumir mensagens longas.
logging.pyPadroniza logs estruturados com structlog.

Guardrail não é censura genérica. É contrato de segurança: privacidade, escopo, formato e ações permitidas.

Data Agent

O Data Agent é o caminho mais prático para estudar um produto real.

O que ele faz

  • Conecta um banco por sessão.
  • Mantém credenciais só em memória.
  • Expõe SQL read-only ao agent.
  • Pode autorizar uma pasta em sandbox Docker.
  • Faz stream de tool calls e tokens ao frontend.

Arquivos principais

  • api/v1/data_agent.py: endpoints.
  • agents/data_agent/agent_data.py: agent.
  • core/db/readonly.py: proteção SQL.
  • core/sandbox/registry.py: fontes por sessão.
Text-to-SQL

O agente de SQL mostra como skills guiam raciocínio especializado.

Fluxo

  1. Recebe pergunta em linguagem natural.
  2. Explora tabelas e schema.
  3. Planeja a query.
  4. Executa SQL read-only.
  5. Explica o resultado.

Peças didáticas

  • AGENTS.md: identidade do agent.
  • skills/schema-exploration/SKILL.md: descobrir estrutura.
  • skills/query-writing/SKILL.md: escrever SQL.
  • chinook.db: banco de exemplo esperado pelo código.

Este é exemplo de estudo. Para produto real, use o Data Agent com conexão por sessão e regras de permissão.

Open Deep Research

Observabilidade

Agente em produção precisa ser observado.

Langfuse

Tracing de chamadas de LLM, metadados, sessões e custo.

Prometheus

Métricas HTTP, duração de stream, tools e inferência.

Grafana

Dashboards provisionados para visualizar latência e saúde.

Sem tracing e métricas, você não sabe se o agent está ajudando, errando, ficando caro ou lento.

Testes e avaliação

Teste mecânica. Avalie qualidade.

Testes

  • tests/integration cobre auth, health, chatbot, deep research, guardrails e text-to-SQL.
  • Fixtures mockam OpenAI e Langfuse.
  • Use uv run pytest tests/.

Evals

  • src/evals avalia respostas usando traces.
  • Métricas são prompts markdown.
  • Inclui relevancy, helpfulness, conciseness, hallucination e toxicity.

Teste responde "quebrou?". Eval responde "continua bom?".

Criando seu primeiro agent

Roteiro prático em 10 passos

  1. Escreva o problema em CONTEXT.md.
  2. Defina usuário, dados, limites e primeira tarefa útil.
  3. Escolha a camada: LangGraph, Deep Agents ou LangChain simples.
  4. Crie src/app/agents/meu_agent/.
  5. Escreva system.md com papel, regras e placeholders.
  6. Implemente tools mínimas e seguras.
  7. Crie DTOs em api/v1/dtos.
  8. Crie rota com auth, rate limit, logging e tratamento de erro.
  9. Registre o router em api.py e o rate limit em config.py.
  10. Teste import, endpoint, fluxo feliz e erro esperado.

Use `.codex/commands/new-agent.md` como receita de scaffold.

Checklist final

Antes de chamar de pronto

Produto

  • O problema está claro?
  • A primeira tarefa é pequena e útil?
  • As fontes de dados estão definidas?
  • O agent sabe quando dizer que não sabe?
  • Há guardrails para dados sensíveis?

Engenharia

  • Sem secrets hardcoded.
  • Rotas com rate limit.
  • Logs estruturados.
  • LLM com tracing.
  • Testes e validação manual.
